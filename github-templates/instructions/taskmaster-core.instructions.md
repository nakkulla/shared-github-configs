# Taskmaster 핵심 워크플로우

## 🎯 핵심 원칙

- **한국어 우선**: 모든 작업, 주석, 문서는 한국어로 작성
- **빠른 구현**: 핵심 기능만 집중, 과도한 완벽주의 금지
- **큰 작업 단위**: 메인 작업 하나씩 순차적으로 완료
- **완료 시 브리핑**: 작업 완료 후 상세 브리핑 + 다음 진행 확인
- **Gemini 모델 필수**: 비용 효율성을 위해 Gemini 사용

## 📝 간결한 PRD 작성 가이드

### ✅ PRD 작성 원칙
- **핵심 기능만**: 복잡한 기능보다 필수 기능 위주
- **빠른 구현**: 1-2주 내 완성 가능한 범위
- **명확한 목표**: 애매한 표현 금지, 구체적 기능 명시

### 📋 PRD 템플릿 (간결 버전)
```markdown
# 프로젝트명

## 목표
- 핵심 문제 해결: [1-2줄로 간단 설명]
- 주요 사용자: [타겟 사용자]

## 핵심 기능 (3-5개만)
1. **기능1**: [구체적 설명]
   - 예시: 사용자 로그인/로그아웃
2. **기능2**: [구체적 설명]  
   - 예시: 데이터 목록 조회/검색
3. **기능3**: [구체적 설명]
   - 예시: 데이터 생성/수정/삭제

## 기술 스택
- Frontend: [React/Vue/etc]
- Backend: [Node.js/Python/etc] 
- Database: [MySQL/MongoDB/etc]

## 제외 사항
- 고급 보안 기능
- 복잡한 권한 시스템
- 실시간 알림
```

### 🚀 빠른 구현을 위한 Task 구성 전략

#### 1. MVP 우선 접근법
```
큰 작업 1: 기본 설정 및 인프라
├── 1.1 프로젝트 초기화
├── 1.2 데이터베이스 설정
└── 1.3 기본 API 구조

큰 작업 2: 핵심 기능 구현
├── 2.1 사용자 인증
├── 2.2 메인 데이터 CRUD
└── 2.3 기본 UI 구성

큰 작업 3: 통합 및 배포
├── 3.1 통합 테스트
├── 3.2 기본 배포 설정
└── 3.3 문서화
```

#### 2. 복잡도 제한 규칙
- **큰 작업**: 복잡도 8점 이하로 제한
- **하위 작업**: 복잡도 5점 이하로 제한
- **8점 초과 시**: 반드시 더 작은 단위로 분해

#### 3. 시간 기반 제한
- **큰 작업**: 최대 1-2일 소요
- **하위 작업**: 최대 2-4시간 소요
- **초과 예상시**: 작업 범위 축소 또는 분해

## 🚀 필수 도구 (핵심만)

### 프로젝트 관리
```typescript
initialize_project              // 프로젝트 초기화
parse_prd                      // PRD → 작업 생성  
get_tasks                      // 작업 목록 조회
get_task --id=X               // 특정 작업 상세 (여러 ID 가능: --id=1,2,3)
next_task                     // 다음 작업 추천
```

### 작업 실행
```typescript
set_task_status --id=X --status="pending|in-progress|done"
expand_task --id=X --research     // 복잡한 작업 → 하위 작업 분해
update_subtask --id=X.Y --prompt="진행상황"
analyze_project_complexity --research
```

### 모델 설정 (⚠️ 필수)
```typescript
models --setMain="gemini-2.5-flash-preview-04-17"  // 초기화 후 즉시 실행
```

## 📋 핵심 워크플로우

### 1단계: 프로젝트 시작 (빠른 설정)
```typescript
// 1. 초기화 및 모델 설정
initialize_project
models --setMain="gemini-2.5-flash-preview-04-17"  // 필수!

// 2. 간결한 PRD 작성 (.taskmaster/docs/prd.txt)
// - 핵심 기능 3-5개만
// - 복잡한 기능 제외
// - 빠른 구현 가능한 범위만

// 3. 작업 생성 및 분석 (8-12개 작업 목표)
parse_prd --numTasks="10"
analyze_project_complexity --research --threshold=6

// 4. 브리핑 준비
get_tasks
complexity_report
```

**⚠️ 중요**: 
- PRD는 **핵심 기능만** 포함 (완벽함보다 빠른 실행)
- 작업 수는 **8-12개**로 제한 (너무 많으면 복잡해짐)
- 복잡도 **6점 이상**만 확장 검토

### 2단계: 빠른 구현 전략

#### 🎯 작업 선택 기준 (빠른 구현)
- **MVP 우선**: 최소 기능 제품부터
- **의존성 우선**: 다른 작업의 기반이 되는 것부터
- **간단한 것부터**: 복잡도 낮은 것 우선 완료

#### 🔄 빠른 작업 진행 패턴
```typescript
// 1. 작업 시작 (빠른 확인)
next_task
get_task --id=X                    // 작업 상세 확인
set_task_status --id=X --status="in-progress"

// 2. 복잡도 체크 (6점 이상만 분해)
// 복잡도 6점 이상 → expand_task --id=X --research
// 복잡도 5점 이하 → 바로 구현

// 3. 하위 작업 진행 (간단하게)
// - 3개 이하: 한번에 처리
// - 4개 이상: 2개씩 배치 처리

// 4. 최소한의 진행 기록
update_subtask --id=X.Y --prompt="완료: [핵심 내용만]"

// 5. 빠른 완료
set_task_status --id=X --status="done"
```

### 3단계: 간소화된 하위 작업 관리

#### 📊 하위 작업 수에 따른 빠른 전략

**3개 이하 → 한번에 처리**
```typescript
get_task --id=X                    // 모든 하위 작업 확인
// X.1, X.2, X.3 연속으로 빠르게 완료
```

**4개 이상 → 2개씩 배치**
```typescript
// 1차 배치 (첫 2개)
get_task --id=X
// X.1, X.2 완료 후 간단 브리핑

// 2차 배치 (다음 2개)  
// X.3, X.4 완료 후 최종 정리
```

#### 🔍 최소한의 진행 기록
```typescript
update_subtask --id=X.Y --prompt="
✅ 완료: [핵심 결과만]
⏰ 소요시간: [실제 시간]
📝 특이사항: [문제나 개선점]
"
```

## 🔧 트러블슈팅 (빠른 해결)

### 자주 발생하는 문제
1. **작업이 너무 복잡** → 범위 축소 또는 분해
2. **진행이 느림** → 완성도 기준 낮추기
3. **기능이 너무 많음** → 핵심 기능만 선별
4. **막힌 상황** → 우선 간단한 버전으로 구현

### 빠른 대응
```typescript
// 막힌 상황 시 - 즉시 단순화
update_subtask --id=X.Y --prompt="문제: [간단 설명], 해결: [간단한 대안]"
// 또는 작업 범위 축소하여 재시작
```
